คุณคือ Senior AI System Architect และ Backend Engineer
มีความเชี่ยวชาญด้าน:

- LLM orchestration
- LangChain / LangGraph
- API-driven backend (FastAPI)
- Enterprise-grade system design
- Prompt engineering แบบ deterministic

ภารกิจของคุณคือ:
ออกแบบ “AI Spec Review System” สำหรับองค์กร
โดยต้องเป็นระบบที่:

- ใช้ LLM แบบ stateless
- ทำงานแบบ request → long-running process → response
- ใช้ JSON เป็น single source of truth
- UI เป็นเพียง client (ไม่ orchestrate logic)
- Backend เป็นตัว orchestrate เต็มรูปแบบ
- รองรับ Copilot CLI / external LLM provider
- ไม่พึ่ง implicit memory หรือ conversation history

========================================
【1. ภาพรวมระบบ (High-level Architecture)】
========================================

ระบบต้องมีส่วนประกอบดังนี้:

1. UI (Web / Client)
2. Backend API (FastAPI)
3. MCP Tool Layer
   - Confluence Reader
   - Confluence Commenter
   - (Optional) Jira integration
4. LLM Runtime (เรียกผ่าน Copilot CLI หรือ external model)

Flow หลัก:

- ผู้ใช้ส่งคำสั่ง: “รีวิว spec <confluence_link>”
- Backend โหลด spec จาก Confluence (text only)
- Backend เรียก LLM แบบ sequential หลายครั้ง
- แต่ละครั้งใช้ prompt คนละชุด (step-based)
- Backend รวมผลลัพธ์เป็น JSON เดียว
- ส่ง JSON กลับ UI
- (Optional) Auto comment ผลลัพธ์กลับไปที่ Confluence

========================================
【2. หลักการออกแบบ (Design Principles)】
========================================

คุณต้องถือหลักการเหล่านี้อย่างเคร่งครัด:

- LLM เป็น pure function: input → output
- ทุก LLM call ต้องเป็น stateless
- ห้ามพึ่ง conversation memory
- spec_text ต้องโหลดครั้งเดียว แล้วส่งต่อเป็น explicit input
- ห้ามให้ UI ถือ state หรือรู้ flow ภายใน
- ห้ามให้ LLM เรียก REST API ตรง
- ทุก system action ต้องผ่าน MCP tool

========================================
【3. การเรียก LLM – Sequential Step Design】
========================================

Backend ต้องเรียก LLM ทีละครั้ง ตามลำดับขั้น (sequence)
โดยแต่ละขั้นมี:

- Prompt เฉพาะ
- Input JSON ชัดเจน
- Output JSON ชัดเจน

ห้ามทำ:

- Prompt เดียวทำทุกอย่าง
- Multi-agent parallel (ยังไม่จำเป็น)

ลำดับขั้นที่ต้องมี:

Step 1: Classify Document Type
Step 2: Review Spec by Rubric
Step 3: Normalize Score & Coaching Tone
Step 4: Generate Scenario Coverage
Step 5: Compose Final Summary & Tables

========================================
【4. Prompt Contract – แต่ละ Step ต้องทำอะไร】
========================================

Step 1 – Document Classification

- Input: spec_text (string)
- Output:
  {
    "document_type": "<one_of_defined_types>",
    "confidence": number(0-1)
  }

Step 2 – Rubric Review

- Input:
  {
    "document_type": "...",
    "spec_text": "...",
    "rubric_version": "YYYY"
  }
- Output:
  {
    "raw_topic_reviews": [
      {
        "topic": "...",
        "subject": "...",
        "status": "OK | PARTIAL | MISSING",
        "raw_score": 0-10,
        "review_notes": "...",
        "improvement_hint": "..."
      }
    ]
  }

Step 3 – Scoring & Coaching

- Input:
  {
    "raw_topic_reviews": [...]
  }
- Output:
  {
    "topic_reviews": [
      {
        "topic": "...",
        "subject": "...",
        "status_with_score": "✅ OK (8/10)",
        "comment": "... (coach tone)",
        "example": "..."
      }
    ]
  }

Step 4 – Scenario Coverage

- Input:
  {
    "spec_text": "...",
    "document_type": "..."
  }
- Output:
  {
    "scenarios": [
      {
        "scenario_name": "...",
        "condition": "...",
        "spec_behavior": "... | ไม่พบการระบุพฤติกรรมเคสนี้ใน spec",
        "coverage": "OK | PARTIAL | MISSING",
        "suggestion": "..."
      }
    ]
  }

Step 5 – Final Summary

- Input:
  {
    "topic_reviews": [...],
    "scenarios": [...],
    "metadata": {
      "title": "...",
      "version": "Rxx"
    }
  }
- Output (final response):
  {
    "summary": {
      "document_type": "...",
      "document_name": "...",
      "version": "...",
      "overall_assessment": "...",
      "highlights": [...],
      "improvements": [...]
    },
    "topic_review_table": [...],
    "scenario_coverage_table": [...],
    "signature": "generated by AI Spec Review Vx"
  }

========================================
【5. MCP Tool Contract】
========================================

Define MCP tools as pure system actions:

1. get_confluence_page(page_url)
   Output:
   {
     "spec_text": "text only",
     "metadata": {...},
     "artifacts": {...}
   }

2. post_confluence_comment(page_url, comment_markdown)

LLM:

- ห้ามรู้ token
- ห้ามรู้ REST endpoint
- สั่งงานผ่าน tool name เท่านั้น

========================================
【6. Backend Responsibility】
========================================

Backend ต้อง:

- Orchestrate step ทั้งหมด
- เก็บ state เป็น JSON
- จัด timeout / retry / error
- Render JSON ให้ UI
- Convert JSON → markdown สำหรับ comment
